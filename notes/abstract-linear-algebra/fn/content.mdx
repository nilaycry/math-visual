at some point, solving problems in linear algebra becomes almost automatic. you read a statement,
the relevant definitions come to mind, and the path forward is mostly clear. I'm not there yet
across the board, but I've crossed that threshold on enough topics to notice the pattern. it's
not that the problems get easier. it's that you stop having to think about how to start.

when you're starting out, that threshold feels far away. so here are the strategies I've found
most useful while getting there.

---

### write out the definitions first

this sounds obvious but it's the move that actually works. before doing anything else, write
down every definition relevant to what you're proving.

if the problem is about a linear map, write $T(u + v) = T(u) + T(v)$ and $T(av) = aT(v)$. if
it's about a subspace, write what a subspace is: non-empty, closed under addition, closed under
scalar multiplication. if it involves a basis, write out both conditions: spanning and linear
independence.

do not paraphrase. write the actual conditions. the proof usually follows from staring at them
long enough.

---

### know what your target looks like

before you start writing, read the conclusion carefully and ask: what does this thing look like
structurally?

if you're proving something is a subspace, you need to show three things. if you're proving a
map is linear, you need to verify two conditions. if you're proving a set spans $V$, you need
to show any vector in $V$ can be written as a linear combination of the set. know the shape of
the argument before you start filling it in.

this is different from knowing the proof. you might not know the proof yet. but knowing the
structure of what you're trying to build gives you the skeleton, and skeletons are easier to
fill in than blank pages.

---

### try a specific example first

when a statement feels abstract, pick a concrete case and check it by hand.

take $V = \mathbb{R}^2$, pick a specific linear map, a specific subspace, whatever the problem
involves, and verify the statement holds there. don't just confirm it works. ask why it works.
what exactly made it true in this case? is that reason specific to $\mathbb{R}^2$, or is it
really about the definitions?

the answer to that question is usually the proof. or at least the key step.

---

### check what type everything is

every object in linear algebra lives somewhere: in $V$, in $W$, in $\mathbf{F}$. scalars and
vectors are different things, and adding them doesn't make sense.

when a step feels stuck or wrong, ask: what type is each object here? does this operation even
make sense? sometimes the issue is that you've written something that isn't well-formed. catching
that early is faster than trying to debug an argument built on a type error.

---

### standard structures have standard first moves

a lot of problems in this course are about specific structures, and those structures have
predictable starting points.

**proving something is a subspace.** check three things: $\mathbf{0}$ is in the set, the set
is closed under addition, the set is closed under scalar multiplication. in that order. you're
almost never going to do something different.

**proving a map is linear.** check two things: $T(u + v) = T(u) + T(v)$ and $T(av) = aT(v)$.
write them out and see if the definition of $T$ makes them obvious.

**proving something is a basis.** check spanning and linear independence. for spanning, show an
arbitrary vector can be written as a linear combination. for independence, show the only solution
to $a_1 v_1 + \cdots + a_n v_n = \mathbf{0}$ is $a_1 = \cdots = a_n = 0$.

**using rank-nullity.** if you know two of the three quantities, you know the third:

$$\dim V = \dim\,\text{null}\,T + \dim\,\text{range}\,T$$

---

### the four proof patterns

most proofs use one of four patterns. worth knowing them automatically.

**direct.** assume the hypothesis, derive the conclusion. the default. try this first.

**contrapositive.** to prove "if $P$ then $Q$," prove instead "if not $Q$ then not $P$." the
two are logically equivalent. useful when the contrapositive is cleaner to work with than the
original.

**contradiction.** assume what you want to prove is false, then derive a contradiction. useful
when you can't construct the conclusion directly but know what goes wrong if it fails.

**induction.** for statements about natural numbers or finite-dimensional spaces. prove the base
case, then prove that if it holds for $n$ it holds for $n + 1$.

---

### when you're stuck

first, try a small example. pick the simplest non-trivial case and trace through it by hand.
ask what made it work, and whether that reason depends on anything specific to the example or
just the definitions.

if I'm stuck on a specific step mid-proof, I write the rough argument anyway. sometimes you
can see exactly what you need to fill in once the structure is on paper, even if the middle is
blank. a partial proof with a clearly marked gap is a lot easier to finish than an empty page.

if the confusion isn't about a specific step but about whether you understand what's going on
at all, that's a different problem. the right move is to go back to the last thing that felt
solid and work forward from there. usually that thing is a definition, and usually there's one
condition in it you haven't really looked at yet.
